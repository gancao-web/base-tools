# Base Tools Feedback

当用户正在使用 `base-tools` skill，且出现以下问题之一时，不要只是在原地继续来回修改；应主动进入反馈流程：

- 未命中：库里明明有相关能力，但 AI 没推荐出来
- 不匹配：推荐存在，但和真实业务场景不符
- 扩展性不足：只差一个参数、配置项、泛型或扩展点就能满足需求
- 明显缺陷：现有函数有 bug、边界问题、类型问题或兼容性异常
- 能力缺失：这是常用场景，但库里没有对应函数

## Flow

### 第一步：分析问题

在触发反馈前，先回顾当前对话并提炼：

1. 用户最初想解决的场景是什么
2. 当前命中的 `base-tools` 包和函数 / hooks / HOC / 指令是什么
3. AI 为什么没有给出合适推荐，归因尽量落到以下问题分类之一：
4. 当前问题更像是 skill 路由问题、文档问题、工具库 API 问题，还是能力缺失问题

问题分类说明：

- 未命中：库里明明有相关函数，但 AI 没推荐出来
- 不匹配：推荐的函数存在，但与真实业务需求不符
- 扩展性不足：只要多一个参数、配置项、泛型或回调扩展点就能满足需求
- 明显缺陷：现有函数行为错误、边界条件有问题、类型不对或平台兼容性异常
- 能力缺失：这是常用场景，但库里没有对应函数，只能临时手写

### 第二步：询问用户

使用自然、简洁、不指责用户的方式询问：

```text
我注意到这个需求里，`base-tools` 出现了未命中、不匹配、扩展性不足、明显缺陷或能力缺失中的一种。这通常意味着 skill 路由、文档描述、现有 API，或者工具库能力覆盖本身还有改进空间。

你是否愿意把这个问题反馈给 `base-tools` 的作者？如果你同意，我会先帮你整理一份精简的问题描述，并生成可直接打开的 GitHub Issue 链接。
```

- 如果用户同意，继续下一步
- 如果用户拒绝，尊重用户决定，继续处理当前问题，不再重复提及反馈

### 第三步：整理反馈内容

生成精简、客观、可复现的反馈内容。

标题格式：

```text
[base-tools] [问题类型]：[场景] 下 [函数 / 能力 / 推荐结果] 不符合预期
```

其中 `[问题类型]` 优先使用：`未命中`、`不匹配`、`扩展性不足`、`明显缺陷`、`能力缺失`

正文格式：

```markdown
## 问题描述

[一句话说明用户想做什么，以及当前 skill 推荐或库能力为什么没有满足需求]

## 使用场景

- 项目类型: [TS / Web / React / Vue / uni-app]
- 使用的 skill: [base-tools]
- 涉及的包/导出: [@base-web-kits/base-tools-xxx / 函数名 / hook 名]
- 问题类型: [未命中 / 不匹配 / 扩展性不足 / 明显缺陷 / 能力缺失]

## 复现过程

1. [用户最初的诉求]
2. [第一次推荐或修改，以及结果]
3. [后续几次调整，以及为什么仍不满足预期]

## 期望行为

[用户真正希望 AI 或工具库能够给出的方案]

## 实际行为

[实际推荐结果、文档缺口、API 限制或理解偏差]

## 可能的改进方向

[基于当前对话，说明更偏向哪种改进：

- 调整 skill 路由或 references，让 AI 更容易命中现有能力
- 补文档 / `catalog.md` / `lookup.md`，减少漏推荐
- 给现有函数增加参数、选项、回调或类型扩展点
- 修复函数 bug、边界行为或平台兼容问题
- 新增一个常用函数 / hook / HOC / 指令]
```

整理原则：

- 精简：只保留关键信息，不复述整段对话
- 客观：描述事实，不放大情绪
- 可操作：让维护者能快速理解并定位问题

### 第四步：生成链接

优先生成 GitHub Issues 新建页链接，并将标题与正文通过 URL 参数预填：

```text
https://github.com/gancao-web/base-tools/issues/new?title={编码后的标题}&body={编码后的正文}
```

可使用如下 Python 示例生成链接：

```python
import urllib.parse

title = """在此填入标题"""
body = """在此填入正文"""

params = urllib.parse.urlencode(
    {
        "title": title,
        "body": body,
    },
    quote_via=urllib.parse.quote,
)

url = f"https://github.com/gancao-web/base-tools/issues/new?{params}"
print(url)
```

把链接、标题和正文原文展示给用户。只有当用户明确要求打开浏览器，且当前环境允许打开外部页面时，再尝试打开该链接：

```text
我已经生成了反馈链接。请打开下面的链接，检查页面中的标题和正文是否准确，确认后提交即可。

如果 GitHub 页面没有自动带出内容，你也可以直接使用下面这份：

标题: [标题内容]

正文:
[正文内容]
```

### 第五步：继续协助

反馈只是补充流程，不应中断当前协作。无论用户是否提交反馈，都继续帮助用户解决眼前问题。
